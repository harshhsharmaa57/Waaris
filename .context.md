## Product in one sentence

Waaris is a non-custodial digital-legacy protocol that coordinates safe, reversible, category-specific posthumous handling of digital assets without putting sensitive plaintext or a master key under a single party's control.

## Current repository state

- Milestones 1, 2, and 3 are implemented: the foundation, isolated authentication, and the first enrollment workflow for Digital Will metadata.
- No trustee, heartbeat, notification, smart-contract, advanced cryptographic, blockchain, or sensitive vault-data behavior exists.
- Engineering workflows are versioned in `skills/`; see `SKILLS.md` before adding, changing, or selecting skills.

## Architectural boundaries

- Clients: React Native mobile app; React + TypeScript web dashboard; SMS/USSD witness fallback.
- Off-chain services: enrollment, heartbeat, notification, witness coordination, crypto-shredding, and a future DP pipeline.
- Data plane: PostgreSQL metadata, Redis heartbeat/TTL data, NATS/Kafka events; encrypted content is off-chain and plaintext must not be persisted by protocol services.
- Trust plane: trustees, witnesses, nominees, oracle network, and external registry/bank interfaces.
- Chain: a future EVM L2 or Cosmos state machine, holding only hashes, commitments, proofs, and state transitions.

## Delivery sequence

1. Establish monorepo tooling, CI, local dependencies, observability baseline, and tests.
2. Build the safe MVP: enrollment metadata, signed heartbeats, notifications, and a non-ZK single-witness local flow.
3. Add accessible dashboard/mobile interfaces and offline/sync foundations.
4. Add contracts and testnet workflow only after the platform baseline is stable.
5. Introduce MPC, ZK, differential privacy, and public storage only with dedicated specifications, threat models, audits, and legal review.

## Foundation implementation

- Go is a multi-module workspace: `platform` supplies infrastructure-only health handlers; enrollment, heartbeat, notification, and witness-coordination are independently buildable service modules.
- The web dashboard is a React + TypeScript + Tailwind/Vite shell with no workflow or data access implementation.
- `docker-compose.yml` provides PostgreSQL, Redis, Mailpit, migrations, all Go shells, and the web dashboard. Docker image verification awaits a running local Docker daemon.
- GitHub Actions Compose validation injects placeholder-only CI environment values for mandatory secrets (`POSTGRES_PASSWORD`, `AUTH_JWT_SECRET`) so interpolation succeeds without weakening local or production secret requirements.
- `infra/migrations` is the only database change path; its initial migration creates an otherwise empty `waaris` schema.

## Authentication implementation

- `services/auth` owns users, password verification, JWT access-token issuance/validation, opaque refresh-token rotation, and `/v1/users/me` profile endpoints.
- Passwords use bcrypt cost 12. JWT access tokens are HS256-signed, issuer-bound, and default to 15 minutes. Refresh tokens are 32 random bytes, rotate on use, and are persisted only as SHA-256 hashes for 30 days.
- Authentication is separate from enrollment so future identity/DID work does not couple to account credentials. All profile endpoints require a Bearer access token.

## Enrollment implementation

- `services/enrollment` owns Digital Will metadata only. It requires a Bearer access token signed with the same JWT secret/issuer as `services/auth`, but it does not depend on auth internals or own any account credentials.
- Each user can have exactly one non-deleted active will. Current will state lives in `waaris.digital_wills`; every create/update appends an immutable snapshot in `waaris.will_versions` plus a consent audit event in `waaris.consent_records`.
- Supported will states are `draft` and `published`. Each write increments the version number, stores the latest dormancy and grace periods, records the accepted policy version, and snapshots normalized release-category preferences (`financial`, `private`, `community_shareable`).
- `DELETE /api/v1/will` is a soft delete of the active aggregate so a user can recreate a new will later without losing historical version and consent evidence until the owning account itself is deleted.

## Non-negotiable safety constraints

- Never store plaintext sensitive content, biometric templates, whole user keys, trustee shares, OTPs, or credentials in logs, database metadata, or chain state.
- Inactivity alone must never cause execution; witness verification and a liveness override/grace period are mandatory.
- Protocol execution provides evidence for legal processes; it does not transfer legal title or funds.
- Treat India-specific legal statements as review items for qualified counsel, not implementation authorization.
